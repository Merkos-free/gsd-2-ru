# Справочник команд

## Команды сеанса

| Команда | Описание |
|---------|-------------|
| `/gsd` | Пошаговый режим — выполнение одного модуля за раз, пауза между каждым |
| `/gsd next` | Режим явного шага (аналогично `/gsd`) |
| `/gsd auto` | Автономный режим — исследование, планирование, выполнение, фиксация, повторение |
| `/gsd quick` | Выполните быструю задачу с гарантиями GSD (атомарные фиксации, отслеживание состояния) без полных затрат на планирование |
| `/gsd stop` | Остановите автоматический режим корректно |
| `/gsd steer` | Жесткое управление плановыми документами во время исполнения |
| `/gsd discuss` | Обсудить архитектуру и решения (работает вместе с автоматическим режимом) |
| `/gsd status` | Панель мониторинга прогресса |
| `/gsd queue` | Ставьте в очередь и меняйте порядок будущих этапов (безопасно в автоматическом режиме) |
| `/gsd capture` | Захват мыслей по принципу «выстрелил и забыл» (работает в автоматическом режиме) |
| `/gsd triage` | Вручную запустить сортировку ожидающих захватов |
| `/gsd forensics` | Посмертное расследование сбоев в автоматическом режиме — структурированный анализ первопричин с проверкой журналов |
| `/gsd cleanup` | Очистка файлов состояния GSD и устаревших рабочих деревьев |
| `/gsd visualize` | Открыть визуализатор рабочего процесса (прогресс, объемы работ, метрики, временная шкала) |
| `/gsd export --html` | Создание автономного отчета HTML для текущего или завершенного этапа |
| `/gsd export --html --all` | Создание ретроспективных отчетов для всех этапов одновременно |
| `/gsd update` | Обновите GSD до последней версии во время сеанса |
| `/gsd knowledge` | Добавьте постоянные знания проекта (правило, шаблон или урок) |
| `/gsd help` | Справочник команд по категориям с описаниями всех подкоманд GSD |

## Конфигурация и диагностика

| Команда | Описание |
|---------|-------------|
| `/gsd prefs` | Выбор модели, таймауты, потолок бюджета |
| `/gsd mode` | Переключение режима рабочего процесса (одиночный/командный) с согласованными настройками по умолчанию для этапа IDs, поведения коммитов git и документации |
| `/gsd config` | Перезапустите мастер настройки поставщика (поставщик LLM + клавиши инструментов) |
| `/gsd keys` | API менеджер ключей — список, добавление, удаление, проверка, ротация, врач |
| `/gsd doctor` | Проверки работоспособности во время выполнения (7 проверок) с автоматическим исправлением распространенных проблем, связанных с повреждением состояния |
| `/gsd skill-health` | Панель мониторинга жизненного цикла навыков — статистика использования, показатели успеха, тенденции токенов, предупреждения об устаревании |
| `/gsd skill-health <name>` | Подробный просмотр одного навыка |
| `/gsd skill-health --declining` | Показывать только навыки, отмеченные для снижения производительности |
| `/gsd skill-health --stale N` | Показать навыки, неиспользованные в течение N+ дней |
| `/gsd hooks` | Показать настроенные перехватчики после устройства и перед отправкой |
| `/gsd run-hook` | Вручную активировать определенный хук |
| `/gsd migrate` | Перенос каталога `.planning` версии 1 в формат `.gsd` |

## Управление этапами

| Команда | Описание |
|---------|-------------|
| `/gsd new-milestone` | Создайте новую веху |
| `/gsd skip` | Запретить отправку устройства в автоматическом режиме |
| `/gsd undo` | Вернуть последний завершенный модуль |
| Веха парка | Доступно через мастер `/gsd` → «Вехи действий» → «Парковка» |
| Отпарковать веху | Доступно через мастер `/gsd` → «Веховые действия» → «Отменить парковку» |
| Отменить этап | Доступно через мастер `/gsd` → «Вехи действий» → «Отменить» |

## Параллельная оркестровка

| Команда | Описание |
|---------|-------------|
| `/gsd parallel start` | Проанализируйте право на участие, подтвердите и начните работу |
| `/gsd parallel status` | Показать всех работников с указанием состояния, прогресса и стоимости |
| `/gsd parallel stop [MID]` | Остановить всех рабочих или рабочего определенной вехи |
| `/gsd parallel pause [MID]` | Приостановить всех рабочих или конкретного |
| `/gsd parallel resume [MID]` | Резюме приостановленных работников |
| `/gsd parallel merge [MID]` | Объединить завершенные этапы вернуться на главную |

Полную документацию см. в разделе [Параллельная оркестровка](./parallel-orchestration.md).

| Ярлык | Действие |
|----------|--------|

| `Ctrl+Alt+V` | Переключить транскрипцию голоса |
| `Ctrl+Alt+B` | Показать фоновые процессы оболочки |

> **Примечание.** В терминалах без поддержки протокола клавиатуры Kitty (macOS Terminal.app, JetBrains IDEs) вместо сочетаний клавиш `Ctrl+Alt` отображаются резервные варианты команд с косой чертой.
>
> **Совет:** Если `Ctrl+V` перехватывается вашим терминалом (например, Warp), используйте `Alt+V` вместо этого для вставки изображения в буфер обмена.

## Безголовый режим

`gsd headless` запускает команды `/gsd` без TUI — предназначено для CI, заданий cron и автоматизации с использованием сценариев. Он порождает дочерний процесс в режиме RPC, автоматически отвечает на интерактивные запросы, обнаруживает завершение и завершает работу с значимыми кодами завершения.

```bash
# Run auto mode (default)
gsd headless

# Run a single unit
gsd headless next

# Instant JSON snapshot — no LLM, ~50ms
gsd headless query

# With timeout for CI
gsd headless --timeout 600000 auto

# Force a specific phase
gsd headless dispatch plan

# Create a new milestone from a context file and start auto mode
gsd headless new-milestone --context brief.md --auto

# Create a milestone from inline text
gsd headless new-milestone --context-text "Build a REST API with auth"

# Pipe context from stdin
echo "Build a CLI tool" | gsd headless new-milestone --context -
```

| Флаг | Описание |
|------|-------------|
| `--timeout N` | Общий таймаут в миллисекундах (по умолчанию: 300000/5 минут) |
| `--max-restarts N` | Автоматический перезапуск при сбое с экспоненциальной задержкой (по умолчанию: 3). Установите 0, чтобы отключить |
| `--json` | Транслируйте все события как JSONL на стандартный вывод |
| `--model ID` | Переопределить модель для безголового сеанса |
| `--context <file>` | Контекстный файл для `new-milestone` (используйте `-` для стандартного ввода) |
| `--context-text <text>` | Встроенный контекстный текст для `new-milestone` |
| `--auto` | Переход в автоматический режим после создания вехи |

**Коды выхода:** `0` = завершено, `1` = ошибка или тайм-аут, `2` = заблокировано.

В качестве позиционного аргумента работает любая подкоманда `/gsd` — `gsd headless status`, `gsd headless doctor`, `gsd headless dispatch execute` и т. д.

### `gsd headless query`

Возвращает один объект JSON с полным снимком проекта — без сеанса LLM, без дочернего объекта RPC, мгновенный ответ (~ 50 мс). Это рекомендуемый способ проверки состояния GSD для оркестраторов и сценариев.

```bash
gsd headless query | jq '.state.phase'
# "executing"

gsd headless query | jq '.next'
# {"action":"dispatch","unitType":"execute-task","unitId":"M001/S01/T03"}

gsd headless query | jq '.cost.total'
# 4.25
```

**Схема вывода:**

```json
{
  "state": {
    "phase": "executing",
    "activeMilestone": { "id": "M001", "title": "..." },
    "activeSlice": { "id": "S01", "title": "..." },
    "activeTask": { "id": "T01", "title": "..." },
    "registry": [{ "id": "M001", "status": "active" }, ...],
    "progress": { "milestones": { "done": 0, "total": 2 }, "slices": { "done": 1, "total": 3 } },
    "blockers": []
  },
  "next": {
    "action": "dispatch",
    "unitType": "execute-task",
    "unitId": "M001/S01/T01"
  },
  "cost": {
    "workers": [{ "milestoneId": "M001", "cost": 1.50, "state": "running", ... }],
    "total": 1.50
  }
}
```

## MCP Режим сервера

`gsd --mode mcp` запускает GSD как сервер [Протокол контекста модели] (https://modelcontextprotocol.io) через стандартный ввод/вывод. При этом все инструменты GSD (чтение, запись, редактирование, bash и т. д.) будут доступны внешним клиентам AI — Claude Desktop, VS Code Copilot и любому MCP-совместимому хосту.

```bash
# Start GSD as an MCP server
gsd --mode mcp
```

Сервер регистрирует все инструменты из сеанса агента и сопоставляет запросы MCP, `tools/list` и `tools/call` определениям инструментов GSD. Он курсирует до закрытия транспорта.

## Обновление во время сеанса

`/gsd update` проверяет npm на наличие более новой версии GSD и устанавливает ее, не выходя из сеанса.

```bash
/gsd update
# Current version: v2.28.0
# Checking npm registry...
# Updated to v2.29.0. Restart GSD to use the new version.
```

Если он уже обновлен, он сообщает об этом и не предпринимает никаких действий.

## Экспорт

`/gsd export` генерирует отчеты о проделанной работе.

```bash
# Generate HTML report for the active milestone
/gsd export --html

# Generate retrospective reports for ALL milestones at once
/gsd export --html --all
```

Отчеты сохраняются в папке `.gsd/reports/` с возможностью просмотра в папке `index.html`, которая ссылается на все созданные снимки.
